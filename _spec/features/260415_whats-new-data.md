# What's New Data

## Summary

Phase 1 of the "What's New" feature: define and implement the data layer
needed to identify and describe content that has changed in the underlying
Git repository since a user's previous visit. This phase establishes a
browser cookie format, a server-side data structure that summarizes changes
relative to the cookie, and the mechanism to derive that structure from the
Git history of the vault. No user-facing UI is built in this phase.

## Goals

- Define the cookie shape used to record a user's last-visit state
- Define a server-side "what's new" data structure derived from the cookie
- Define and implement the Git-backed mechanism that produces the data structure
- Provide a foundation that a future Phase 2 UI can consume without rework

## Non-Goals

- Any user-facing UI: no "what's new" button, no summary page, no change bars
- Per-element/in-document diff rendering
- Per-user accounts or server-side history beyond what Git already provides
- Notifications, email digests, or RSS feeds
- Tracking of read/unread state at element granularity

## User Stories

1. **Returning visitor state**: As a returning user, my browser holds a small
   cookie recording the date and Git commit at the time of my last visit, so
   the server can later tell me what has changed.
2. **Server-side summary**: As the application, given a request that carries a
   "last visit" cookie, I can produce a structured summary of which documents
   in the vault have been added, modified, renamed, or deleted since that
   commit.
3. **Cold start**: As a first-time visitor with no cookie, the system records
   a fresh last-visit cookie and treats the visit as having no prior baseline.
4. **Cookie refresh**: As a returning user, after the server has computed the
   change summary for my visit, my cookie is updated to the current commit and
   visit time so the next visit compares against this point.
5. **Stale or unknown commit**: As a returning user whose recorded commit is
   no longer present in the repository (e.g. history rewritten, branch
   removed), the system gracefully falls back to a sensible baseline rather
   than erroring.

## Functional Requirements

### Cookie

- Name and scope of the cookie are defined in this spec
- Cookie payload contains at minimum:
  - `last_visit_date` — ISO 8601 timestamp of the user's prior visit
  - `last_git_commit` — full SHA of the commit that was current at that visit
- Cookie is set on first visit and refreshed on each subsequent visit, after
  the change summary for that visit has been computed
- Cookie has a defined lifetime and SameSite policy appropriate for a
  self-hosted app
- Cookie value is opaque to the client (no need for the browser to read it)

### What's New Data Structure

- A server-side struct/map representing changes between the cookie's
  `last_git_commit` and the current `HEAD`
- Includes:
  - `from_commit`, `to_commit`, `from_date`, `to_date`
  - A list of changed files, each entry containing:
    - vault-relative path
    - change kind: `:added`, `:modified`, `:renamed`, `:deleted`
    - previous path (for renames)
    - last commit SHA, author, and timestamp that touched the file in the range
  - A summary count by change kind
- Only files within the served vault directory are reported
- Only file types that are surfaced by the app (e.g. markdown, PDFs, images)
  are reported; other files are filtered out
- Deterministic ordering (e.g. by most recent change first)

### Git-backed Generation

- The data structure is generated by inspecting the Git repository that backs
  the vault directory
- Works against the on-disk repository; no external Git server required
- Handles the following cases:
  - Cookie commit equals current HEAD → empty change set
  - Cookie commit is an ancestor of HEAD → normal range diff
  - Cookie commit is unknown to the repository → fall back to "no baseline"
  - Repository is not a Git repo → feature degrades to "no changes available"
- Performance: generation should be cheap enough to run on each page request
  for a typical small vault; if needed, results may be cached keyed by
  `(from_commit, to_commit)`

### Integration Points

- A single function/module is the canonical entry point: given the request's
  cookie (or its absence), it returns the What's New data structure and the
  cookie value to write back on the response
- The Phoenix layer is responsible for reading the cookie from the connection
  and writing the refreshed cookie to the response
- The data structure is exposed in a form that LiveView and plain controllers
  can both consume in Phase 2

## UI/UX Requirements

- None in Phase 1. No visible UI changes.
- The feature must not regress existing pages: cookie writes must not break
  caching, and computation must not noticeably slow page loads.

## Open Questions

- Should the cookie be signed/encrypted using Phoenix's session cookie
  machinery, or stored as a plain cookie since its contents are non-sensitive?  Answer: use the Phoenix mechanism
- What is the appropriate cookie lifetime — session-only, 30 days, 1 year?  Answer: 30 days - after that every visitor is treated as a first-time visitor
- Should the "current commit" be recomputed per request, or cached and
  refreshed on a timer / file-watch event?  Answer: timer is fine, or when there is a pub-sub event
- For renames, do we treat them as a single `:renamed` entry or as a
  `:deleted` + `:added` pair?  Answer: single rename
- How should non-fast-forward history (force pushes, rebases) be detected and
  signaled to the consumer of the data structure?  Answer: ignore - just react to git commits
- Should the data structure include per-file change magnitude (lines
  added/removed), or is presence-of-change sufficient for Phase 2?  Answer: probably the former.  
- Where should the canonical module live in the codebase
  (`Docpub.WhatsNew`, `Docpub.Vault.History`, etc.)?  Answer: Docpub.WhatsNew seems good.

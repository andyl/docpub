# Design: AI Chat

**Date:** 2026-04-16
**Status:** Draft

---

## Goal

Let users ask natural-language questions about their vault ("when did I last go
to Colorado?", "summarize the work I've done on the van brakes") and get
sourced answers grounded in their documents.

## Non-goals

- Generating or editing vault documents from chat.
- Multi-user / shared conversations.
- Real-time vault indexing (re-index can be manual or on-startup).

---

## UI Layout

A new `/chat` route with a two-column content area beside the existing sidebar.

```
┌──────────────────────────────────────────────────────┐
│  Header                                              │
├──────────┬───────────────────┬────────────────────────┤
│ Sidebar  │  Chat             │  Document Preview      │
│ (tree)   │                   │                        │
│          │  ┌─────────────┐  │  Shows the cited doc   │
│          │  │ AI: answer   │  │  when user clicks a   │
│          │  │ with [1][2]  │  │  citation link.        │
│          │  └─────────────┘  │                        │
│          │  ┌─────────────┐  │  Scrolled/highlighted  │
│          │  │ User: query  │  │  to the relevant       │
│          │  └─────────────┘  │  section.               │
│          │  ┌─────────────┐  │                        │
│          │  │ AI: ...      │  │  When nothing is       │
│          │  └─────────────┘  │  selected, show a list  │
│          │                   │  of "sources used" as   │
│          │  ┌─────────────┐  │  clickable cards.       │
│          │  │ [input]  [>] │  │                        │
│          │  └─────────────┘  │                        │
├──────────┴───────────────────┴────────────────────────┤
```

### Chat column (~45% of content area)

- Scrollable message list, newest at bottom.
- Input area: textarea, enter to send, shift+enter for newline.
- While the AI is working, show a "Searching N documents..." then
  "Thinking..." indicator. Stream tokens as they arrive.
- Each AI message can contain inline citations `[1]`, `[2]` that are
  clickable links.

### Document preview column (~55% of content area)

- Reuses the existing `vault-prose` markdown rendering.
- Clicking a citation loads the referenced document, scrolled to the
  relevant chunk, with the matched passage highlighted.
- When no citation is selected, displays a "Sources" card list showing
  which documents were retrieved for the current answer.
- Clicking a sidebar tree node while in chat mode could load the doc
  in this preview pane rather than navigating away.

### Conversation persistence

Start simple: conversations are session-only (held in LiveView assigns).
Future option: persist to `_chat/` directory as JSON or markdown files,
with a conversation list in the sidebar.

## Retrieval Architecture

This is the core technical challenge. Three approaches worth evaluating,
from simplest to most capable:

### Option A: BM25 (keyword-based retrieval)

BM25 is a term-frequency ranking algorithm — the classic "non-vector RAG"
approach. It scores documents by how well their term frequencies match the
query, with diminishing returns for repeated terms (TF saturation) and a
boost for rare terms (IDF).

**How it works in Docpub:**

1. On startup (or manually), chunk every `.md` file into passages
   (e.g., by heading section or sliding window of ~300 tokens).
2. Build a BM25 index in SQLite using its FTS5 extension, which has
   built-in BM25 ranking (`SELECT *, bm25(docs_fts) FROM docs_fts
   WHERE docs_fts MATCH ?`).
3. At query time, run the user's question through FTS5, take top-N
   chunks, and feed them to the LLM as context.

**Pros:** Simple, no embeddings API calls, no vector math, fast, works
well for factual lookups ("when did I go to Colorado"). SQLite FTS5 is
mature and built into Erlang's `:sqlite3` driver.

**Cons:** Keyword mismatch problem — if the user says "van brakes" but
the document says "brake pads on the Sprinter", BM25 may miss it.

**Elixir libraries:** `exqlite` or `ecto_sqlite3` for SQLite access.

### Option B: Vector embeddings (semantic search)

Embed each chunk into a vector using an embeddings API (e.g., Anthropic's
`voyage-3` or OpenAI's `text-embedding-3-small`), store in SQLite with
the `sqlite-vec` extension, and do cosine-similarity search at query time.

**How it works in Docpub:**

1. Chunk documents as in Option A.
2. Call the embeddings API for each chunk, store the vector in a
   `sqlite-vec` virtual table alongside the chunk text and metadata.
3. At query time, embed the user's question, query for top-N nearest
   vectors.

**Pros:** Handles semantic similarity ("van brakes" matches "Sprinter
brake pads"). Better for open-ended questions.

**Cons:** Requires an external API for embeddings (cost, latency,
network dependency). `sqlite-vec` is newer and requires compiling a
C extension. Re-embedding on document change adds complexity.

**Elixir libraries:** `exqlite` + `sqlite-vec` NIF, or `ecto_sqlite3`.
For embeddings: `Req` to call the API directly.

### Option C: Hybrid (BM25 + vectors)

Use both retrieval methods and merge results via Reciprocal Rank Fusion
(RRF) or a simple union-and-rerank strategy. This is what production RAG
systems typically converge on.

**How it works:**

1. Run BM25 search → top 20 candidates.
2. Run vector search → top 20 candidates.
3. Merge, deduplicate, rerank (RRF or LLM-based rerank).
4. Take top-N chunks for the LLM context.

**Pros:** Best retrieval quality — BM25 catches exact keyword matches
that vectors miss, vectors catch semantic matches that BM25 misses.

**Cons:** Most complex. Two indexes to maintain.

### Option D: PageIndex (LLM-navigated, vectorless)

PageIndex is a vectorless RAG technique where the LLM navigates documents
like a human would — using a table-of-contents / index structure to
decide which sections to read, then drilling into those sections for
detail. No embeddings, no vector math.

**How it works:** Build a hierarchical index of the corpus (document
titles, heading structures, summaries). The LLM reads the index, decides
which pages/sections are relevant, requests those chunks, and iterates
until it has enough context to answer.

**Pros:** No embedding API calls, handles very long documents well,
leverages the LLM's own reasoning to navigate. Can surface connections
that keyword and vector search miss.

**Cons:** Core use case is very long documents (which our vault doesn't
have — most are short Obsidian notes). Multiple LLM round-trips per
query increase latency and cost. More complex orchestration logic.

**Assessment:** Interesting technique but not a natural fit for Docpub's
vault of many short documents. Worth revisiting if the vault grows to
include long-form content.

**Reference:** https://medium.com/@omtita.codes/vectorless-rag-letting-an-llm-navigate-documents-like-a-human-d5834ad518ec

### Recommendation

**Start with Option A (BM25 via SQLite FTS5).** It requires no external
API for retrieval, is fast, and handles the majority of factual vault
queries well. The FTS5 index is trivial to rebuild on startup since the
vault is file-based.

Then evaluate whether semantic gaps are a real problem in practice. If
they are, add vector search (Option C hybrid) as a second pass.

## Chunking Strategy

Documents need to be split into retrievable passages. Options:

1. **By heading section**: Split on `## ` boundaries. Natural for
   Obsidian-style documents. Preserves context well.
2. **Sliding window**: Fixed token window (e.g., 300 tokens) with
   overlap (e.g., 50 tokens). More uniform chunk sizes.
3. **Paragraph-based**: Split on double newlines. Simple but can
   produce very short or very long chunks.

Recommendation: **heading-based chunking** as primary strategy, with
a fallback to sliding window for documents without headings. Each
chunk stores: file path, heading path (e.g., "Van > Brakes > April
Service"), line range, and raw text.

## LLM Integration

### API

Use the Anthropic API via `Req` (consistent with the project's HTTP
client convention). The `claude-sonnet-4-20250514` model is a good
balance of quality and speed for RAG answers.

### Prompt structure

```
System: You are a helpful assistant answering questions about a
personal knowledge base. Answer based ONLY on the provided documents.
Cite sources using [1], [2], etc. If the answer isn't in the
documents, say so.

[Retrieved chunks as numbered sources]

User: {question}
```

### Streaming

Use server-sent events from the Anthropic API. In LiveView:

1. User submits question → event handler starts a `Task.async`.
2. Task streams the API response, sending chunks via
   `send(self(), {:chat_delta, chunk})`.
3. `handle_info({:chat_delta, chunk})` appends to the current
   message and LiveView pushes the update to the client.

This gives a responsive typing-indicator UX.

## Open Questions

1. **Index lifecycle**: Rebuild on every app startup? Or persist the
   SQLite DB and incrementally update when files change? (The vault
   file watcher already exists and could trigger re-indexing.)
2. **Chunk size tuning**: How large should chunks be? Larger chunks
   give the LLM more context but reduce the number that fit in the
   context window.
3. **Conversation context**: Should previous messages in the
   conversation influence retrieval? (e.g., "tell me more about that"
   should re-use the prior query's context.)
4. **Cost/privacy**: Vault content is sent to the Anthropic API. Is
   that acceptable? (Anthropic's API doesn't train on customer data,
   but worth noting.)
5. **Auth**: The chat route should sit behind the same `vault_auth`
   pipeline as the doc routes.
